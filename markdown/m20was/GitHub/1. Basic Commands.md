# 1. Basic Commands 📂

This guide covers the core workflow of Git. These are the tools you will use daily to initialize repositories, track changes, save progress, and inspect history.

---

## 🛠️ Configuration & Setup

Before making commits, you should configure your identity. Git uses this information to associate commits with a name and email.

### Set Global Identity
```bash
# Set your global username
git config --global user.name "Your Name"

# Set your global email address
git config --global user.email "your.email@example.com"
```

### Check Configuration
```bash
# List all current configuration settings
git config --list

# Show where configurations are defined (system, global, local)
git config --list --show-origin
```

---

## 🆕 Initializing & Cloning

To start version-controlling files, you either create a new repository from scratch or copy an existing one.

### Initialize a New Local Repository
```bash
# Turn the current directory into a Git repository
git init

# (Optional) Initialize and specify the default branch name as 'main'
git init -b main
```
> [!NOTE]
> Running `git init` creates a hidden `.git` folder in your directory. This is where Git stores all repository metadata and history. **Do not delete it** unless you want to stop tracking the folder.

### Clone an Existing Repository
```bash
# Clone a repository from a URL (GitHub, GitLab, etc.)
git clone <repository-url>

# Clone into a specific folder name
git clone <repository-url> <folder-name>
```

## 🔍 Tracking & Status

Git needs to know which files it should watch and compile into the next save point (commit).

### Check Status
```bash
# Show modified, untracked, and staged files
git status

# Show status in a short/compact format (great for quick checks)
git status -s
```

### Stage Changes (Add to Staging Area)
The staging area is a prep zone where you choose what changes will go into the next commit.
```bash
# Stage a specific file
git add <filename>

# Stage all files in the current directory (including subdirectories)
git add .

# Stage only updated/deleted files (no new untracked files)
git add -u

# Interactive staging (choose specific lines/chunks to stage)
git add -p
```

---

## 💾 Saving Progress (Commits)

A commit is a snapshot of your staged files at a specific point in time. Think of it as a game save.

### Commit Staged Changes
```bash
# Commit with a short, descriptive message
git commit -m "Add authentication module"

# Stage all modified files and commit them in one step (skips git add for already tracked files)
git commit -am "Update landing page layout"

# Modify the most recent commit (fixes messages or adds missed staged changes)
git commit --amend -m "Corrected commit message"
```

---

## 📜 Inspecting History

Reviewing what has been done in the repository.

### View Commit History
```bash
# Show full commit history (press 'q' to exit)
git log

# Show history as a clean, one-line list
git log --oneline

# Show history with a visual text-based branch graph
git log --oneline --graph --all

# Show commits limit to the last N commits
git log -n 5
```

### View Differences
```bash
# Show changes in working directory that are NOT yet staged
git diff

# Show changes that ARE staged but not yet committed
git diff --staged

# Compare differences between two specific commits
git diff <commit-hash-1> <commit-hash-2>
```

---

## 💡 Quick Tips
* **Write Good Commit Messages**: Use the imperative mood (e.g., "Add feature X" instead of "Added feature X" or "Adds feature X").
* **Use `.gitignore`**: Create a file named `.gitignore` at the root of your project to specify files and folders Git should ignore (e.g., `.venv/`, `node_modules/`, system config files like `.DS_Store`, or sensitive secrets).
