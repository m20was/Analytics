# 5. Advanced & Debugging 🚀

This guide covers advanced operations that help you recover lost files, clean up your commit history, cherry-pick work, and manage temporary changes.

---

## 📦 Git Stash (Save Work Temporarily)

Stashing saves your modified, tracked files in a temporary stack, returning your working directory to clean status. This is perfect for when you need to switch branches quickly but aren't ready to commit.

### Basic Stash Commands
```bash
# Save current modifications to the stash stack
git stash

# Save modifications and give the stash a custom descriptive name
git stash save "Work in progress on login layout"
# (or modern syntax)
git stash push -m "Work in progress on login layout"

# Include untracked files in the stash
git stash -u
```

### Retrieving Stashed Changes
```bash
# List all stashed changes
git stash list

# Apply the most recent stash and remove it from the stash stack
git stash pop

# Apply a specific stash (e.g. stash@{2}) but keep it in the stash stack
git stash apply stash@{2}
```

### Deleting Stashes
```bash
# Delete a specific stash from the stack
git stash drop stash@{0}

# Delete all stashes in the stash stack
git stash clear
```

## 🍒 Cherry-Picking (Copying Specific Commits)

Cherry-picking allows you to select a single commit from one branch and apply it to another branch as a new commit.

```bash
# 1. Switch to the branch you want to apply the commit to
git switch main

# 2. Cherry-pick the commit by its hash
git cherry-pick <commit-hash>
```

## 🕵️ Git Reflog (The Ultimate Safety Net)

Did you run a `git reset --hard` and lose a commit? Did you delete a branch by accident?
Git keeps a log of every movement of `HEAD` (the pointer indicating your active branch and commit) for local operations. This is the **Reflog**.

```bash
# View the list of HEAD movements
git reflog
```

The output looks like this:
```text
c3a2b1f HEAD@{0}: reset: moving to HEAD~1
f5e4d3c HEAD@{1}: commit: Add database indexing
a1b2c3d HEAD@{2}: checkout: moving from feature to main
```

### Recovering a Lost Commit:
If you reset back to `a1b2c3d` and lost `f5e4d3c` ("Add database indexing"), you can recover it:
```bash
# Reset your active branch back to the commit before the reset
git reset --hard f5e4d3c
# (or using reflog notation)
git reset --hard HEAD@{1}
```

## ✏️ Interactive Rebasing (Cleaning Commit History)

Interactive rebasing allows you to rewrite commit history before pushing to remotes. You can combine (squash) multiple small commits, edit commit messages, or delete unwanted commits.

```bash
# Start an interactive rebase for the last 4 commits
git rebase -i HEAD~4
```

This opens your default text editor showing a list of commits with commands:
```text
pick a1b2c3d Add feature setup
pick f5e4d3c Fix small typo in css
pick 2b3c4d5 Fix another typo
pick 9z8y7x6 Complete feature implementation
```

### Rebase Commands:
* **`pick`**: Keep the commit as is.
* **`reword`**: Keep the commit but edit the commit message.
* **`edit`**: Pause the rebase to modify files or add changes.
* **`squash`**: Combine this commit with the one above it, merging their commit messages.
* **`drop`**: Remove this commit entirely.

*Save and close the file to run the rebase process. Git will walk you through each step.*

## 🏷️ Tagging

Tags are reference points used to mark specific points in history as important, typically used for software releases (e.g. `v1.0.0`).

### Create Tags
```bash
# Create a lightweight tag (just a pointer)
git tag v1.0.0-lw

# Create an annotated tag (includes creator, date, and message - Recommended)
git tag -a v1.0.0 -m "Release version 1.0.0"
```

### Share Tags
Tags are not automatically pushed to remote servers. You must push them explicitly.
```bash
# Push a specific tag to remote
git push origin v1.0.0

# Push all local tags to remote
git push origin --tags
```
